Knowledge Operations / Agent Memory

讓知識庫每天自己變乾淨。

白天收進來的聊天、文件、工具輸出與專案線索，不應該隔天就散掉。Wiki Agent 的任務，是把它們整理成可追溯、可審核、可被 agent 使用的長期知識。

核心目標 先整理，再記憶；先審核，再放進正式 wiki。
Daily 每天自動更新知識狀態
Review 候選知識先審核再採用
Trace 每段重點都保留來源
5 層 raw / inbox / lint / archive / wiki

這不是筆記系統，是知識營運管線

OpenWiki 的精神是讓 coding agent 不必把整個 repo 背進上下文，而是知道該去 wiki 哪裡查。企業導入 Wiki Agent 時，也可以用同樣邏輯整理文件、SOP、專案紀錄與教學資料。

Capture

收進文件、網址、專案紀錄與工作輸出。先保留來源，不急著變成正式知識。

Intake

把有價值的片段轉成 raw / inbox proposal，讓 Wiki Agent 可以審核。

Curate

檢查來源、日期、可信度、重複、過期、疑似 secret。品質過關才往前。

Publish

正式 wiki 只放整理後的知識。舊資料不直接刪除，而是移到 archive。

Route

讓使用者與 AI 工具讀整理後的摘要，不直接吞原始雜訊，避免越用越亂。

OpenWiki 給我們的三個啟發

這份教材不是要照搬 OpenWiki，而是取它最重要的設計原則：文件要給 agent 用、要能自動更新、要用短引用連到長知識。

01 / Agent-ready Docs

文件不是只給人看的

coding agent 需要知道關鍵邏輯在哪裡、檔案怎麼連、專案習慣是什麼。Wiki 的價值，是讓 agent 可以查到上下文，而不是每次重新猜。

AAGENTS.md / CLAUDE.md 只放指路規則，不塞整本百科。
B真正的內容留在 wiki，讓 agent 需要時再讀。
02 / Daily Refresh

知識會過期，所以要排程

文件第一次生成不難，難的是每天跟著專案、工具和流程改變。夜間維護的目的，是讓知識品質不靠人想到才整理。

A每天掃新資料、補狀態、重建索引。
B用 lint 和 archive 控制雜訊，不讓待整理區變垃圾場。
03 / Proactive Memory

不是記你說過什麼，是整理你工作裡發生什麼

OpenWiki Brains 把多個來源轉成可維護的 wiki。對企業訓練來說，重點是把零散文件、專案紀錄、操作 SOP 與教學素材沉澱成可查的工作記憶。

A先把來源保存清楚，後面才能追溯。
B讓記憶變成檔案系統中的 Markdown，而不是黑箱資料庫。

每天晚上的七個站點

真正重要的是順序。Wiki Agent 不是把所有資料直接寫進正式 wiki，而是先做分流、檢查、封存與重新整理。

1

Harvest

掃描新增文件、工作紀錄、會議整理與教學素材。保留「可能可重用」的片段。

2

Intake

把片段變成 proposal，送進 raw/ 和 inbox/。這一步是候選知識，不是正式知識。

3

Lint

檢查 Source、Source date、Confidence、重複來源、過期項目和疑似 secret。品質問題要先浮出來。

4

Archive

舊的或重複的 top-level raw / inbox 會移到 archive/。不是刪除，而是可回溯地降噪。

5

Index

重建索引與知識圖，讓後續查找不是靠運氣，而是有穩定入口和結構。

6

Eval

用 golden QA 和日常評估確認知識搜尋還能回答問題。知識庫不是只要多，還要答得準。

7

Refresh

更新 quickstart、open questions、maintenance 頁面，讓人和 agent 都看得懂現在狀態。

8

Evolve

如果 backlog 或 metadata 問題變多，產生改善提案，讓負責人判斷要不要升級流程。

分權要清楚，知識才不會壞掉

知識系統最容易失控的地方，是「誰都可以把內容寫進正式知識庫」。這套做法把使用、整理、審核與發布分開。

角色分工

使用者 提出問題、補充資料、確認內容是否可採用。
整理者 把來源分組、補摘要、標示適合的用途。
Wiki Agent 維護 raw / inbox / wiki / archive 的品質。
審核者 確認來源、可信度、敏感資訊與是否能發布。

最重要的治理規則

AI 工具不需要背全部知識。它需要知道去哪裡找、哪些內容已經審核、哪些內容還不能進正式 wiki。

資料狀態處理方式
新來源先進 raw，保留原始路徑或網址。
候選知識進 inbox，等待審核與補 metadata。
正式知識進 wiki，可被團隊和 AI 工具引用。
過期或重複進 archive，不直接刪除。

導入時要看的品質訊號

如果只看「今天新增幾頁」，很容易被假成長騙到。真正要看的，是 backlog、引用品質、重複率與是否能回答問題。

Signal

Inbox backlog

待整理越堆越多，代表系統有收資料，但沒有完成知識整理。這時要做批次分類和優先級整理。

Signal

Metadata issues

缺 Source、日期或 Confidence，代表未來很難判斷這段知識可信不可信。這種資料不能直接升正式 wiki。

Signal

Answer quality

知識庫不是收藏夾。最後要測的是：agent 查得到嗎？答得準嗎？引用來源夠少、夠精準嗎？

可以直接使用的操作句

這些不是要背指令，而是幫你把「收集、整理、審核、查詢」變成固定工作語言。

查狀態 請檢查今天 raw、inbox、wiki、archive 的狀態，列出最需要整理的前三個問題。
收新來源 請把這個網址或檔案先做 intake，保留原始來源，不要直接寫進正式 wiki。
整理候選知識 請把 inbox 裡跟工作流程、工具操作、知識治理有關的項目分組，列出可升正式 wiki 的候選。
夜間維護 每天固定整理新增來源、封存過期項目、檢查 metadata，並回報新增、封存、待審核與索引更新狀態。
