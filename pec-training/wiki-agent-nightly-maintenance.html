Wiki Agent Nightly Maintenance：讓知識庫每天自動收錄、整理與封存

這份教材把本地 Wiki Agent 的升級拆成可落地的知識營運流程：白天讓人和 agent 產生資料，晚上由 Wiki Agent 自動收錄、整理、封存、更新導覽頁，讓知識庫不再只是堆檔案。

整理基礎：本地 Wiki Agent 實作 / OpenWiki 借鏡 / ADLINK 內訓整理 2026-07-15
Course Framing

真正的問題不是「有沒有知識庫」，而是知識庫會不會自己持續變乾淨、變可用。

很多團隊的 wiki 一開始都很漂亮，但過幾週就變成原始資料堆、待整理 inbox、重複頁面、過期 SOP 混在一起。這次做的不是多一篇筆記，而是把 Wiki Agent 變成一個每天會自動巡檢的知識管家。

這次升級在解決什麼痛點

原始問題

知識只進不出

大家一直丟新資料，但沒有人固定整理、封存、補 metadata，最後 wiki 只會越來越亂。

流程問題

聊天紀錄不會沉澱

Codex、其他 agent、Telegram 對話裡其實有很多 know-how，但沒有固定回收機制，隔天就散掉。

治理問題

管理權和寫入權混在一起

如果 orchestrator 既分派工作又直接亂改正式 wiki，系統很快就會失控。

這次解法

把知識營運變成夜間流程

白天收資料，晚上由 Wiki Agent 自動 harvest、lint、archive、refresh，讓知識品質變成制度，不靠人記得。

一句話：這次不是做一個更大的 wiki，而是做一個會自己持續維護的 wiki 系統。

正確分權：小愛調度，Wiki Agent 管知識品質

這次保留了你原本最重要的治理原則：

  • 小愛 負責 agent orchestration、任務分派、讀取摘要後再決定要把 know-how 給誰。
  • Wiki Agent 是唯一負責正式知識庫維護的 agent，管理 raw/、inbox/、wiki/ 的品質。
  • 其他 agent 不直接改正式 wiki，只能送 proposal 或 memory 給 Wiki Agent。
  • 夜間排程 幫忙把白天產生的知識碎片重新整理成可持續使用的內容。
治理重點
讓 orchestrator 更強的方法，不是讓它自己背全部知識，而是讓它知道：什麼知識去哪裡查、什麼時候叫 Wiki Agent 更新、什麼東西值得進長期記憶。

這次升級前後差異

項目 升級前 升級後
入口 多支 script 分散執行 用 wiki-agent.mjs 做統一入口
聊天紀錄回收 沒有固定收錄機制 每天固定掃 Codex archived sessions、agent chat logs、memory docs
清理方式 主要靠人工看 lint 自動 archive 舊的或明顯重複的 top-level raw / inbox
導覽頁 只有 index，狀態不夠即時 新增 quickstart / open-questions / maintenance 三頁每日更新
夜間排程 偏向 lint / index 變成完整 nightly maintenance：harvest、archive、graph、eval、refresh
知識可見度 白天有新東西，但晚上不一定被整理 每天固定巡檢，讓知識庫持續變乾淨、變可讀、變可用

每天晚上的維運流程

  1. Harvest：掃描最近的 Codex archived session、其他 agent 的 chat_history.jsonl、以及 learned_wiki.md / know-how-sops.md。
  2. Intake：把值得留下來的片段轉成新的 raw/ 與 inbox/ proposal，而不是直接改正式 wiki。
  3. Lint：檢查缺 metadata、重複來源、過期 inbox、矛盾標記與疑似 secret。
  4. Archive：把太舊或測試性質的 top-level raw/ / inbox/ 搬進 archive/，不直接刪除。
  5. Refresh：更新 wiki/quickstart.md、wiki/open-questions.md、wiki/maintenance.md。
  6. Graph + Eval：重建知識圖、跑 Golden QA 與日常 eval，確認知識搜尋還能用。
  7. 小愛接手：如果新知識值得變成 capability action，再讓小愛決定要不要分派給其他 agent。
這裡的關鍵不是「自動寫 wiki」，而是「先自動整理 proposal，正式 wiki 仍然由 Wiki Agent 維護」。這樣才不會讓知識庫被未驗證內容污染。

這次實際補上的核心腳本

wiki-agent.mjs

單一入口。之後用 maintain、harvest、status 等子命令呼叫整套流程，不用再記一堆分散腳本。

wiki-agent-harvest.mjs

回收聊天與 agent memory，把真正像 workflow、SOP、routing rule、know-how 的東西送進 Wiki Agent 的 intake 流程。

wiki-agent-archive.mjs

處理 top-level raw/ 和 inbox/ 的舊資料。原則不是刪除，而是可回溯地搬進 archive/。

wiki-agent-refresh.mjs

生成 OpenWiki 味道更重的導覽頁，讓使用者一打開就看到現在狀態、待整理問題和最新維護結果。

wiki-agent-evolve.mjs

原本比較偏 self-evolution audit，現在擴成 nightly maintenance 總控，串起 harvest、archive、lint、graph、eval。

LaunchAgent

把 22:40 的排程實際寫回 macOS LaunchAgents，讓這不是 repo 內的設定，而是每天真的會跑的服務。

這件事對 ADLINK 團隊真正有什麼用

對工程團隊

  • 會議中的 know-how、debug 過程、工具輸出不會只留在聊天視窗裡。
  • RAG / wiki / SOP / skill 的界線會更清楚，不會什麼都混在同一層。
  • 你可以把知識治理變成夜間批次，而不是白天人工補洞。

對內訓與知識傳承

  • 教材產線、prompt 工作法、agent 維運規則可以持續沉澱，而不是每次重講。
  • 新進同事看 quickstart、open-questions、maintenance 就知道知識庫現在狀態。
  • 當 agent 數量增加時，知識品質不會完全靠小愛一個總管硬背。
最實用的導入方式
先把這套做法用在教材產線、PM note、產品知識整理、Telegram / Codex 工作紀錄回收。等運作穩定，再往更多 agent 與部門擴。

如果你要複製這套做法，至少先做到這幾件事

  • 先分權：orchestrator 不直接改正式 wiki，Wiki Agent 才是正式知識維護者。
  • 所有新資料先進 raw/ 或 inbox/，不要跳過 review 直接寫 wiki。
  • 每天至少固定一次自動 maintenance，不然知識庫只會積灰。
  • 保留 archive，不要因為想乾淨就直接刪資料。
  • 把 status、open questions、maintenance 做成人人看得懂的頁面，避免系統變黑箱。
  • 讓小愛只讀摘要與結果，不要把所有 raw memory 都塞給它。
最值得記的一句話
好的知識型 agent，不是知道很多，而是每天都能把新知識收進來、把舊知識整理好、把垃圾知識隔離出去。
