Context Headroom 不是記憶庫，是上下文清理器

Headroom 官方把自己定位為 LLM 應用的 context optimization layer。對 Michael 的本地 Hermes agent 系統來說，價值不在省 token 費用，而是降低長任務爆 context、工具輸出塞滿視窗、交接摘要失焦的機率。

課程大綱

核心問題：長任務不是缺大腦，是上下文太髒

本地 LLM 不太怕 token 成本，但長時間 coding、Wiki 整理、YouTube transcript、Excel 報表與 bot log 會把 context window 塞滿。當 agent 看太多重複輸出，就會忘記真正要做的決策。

Headroom 的定位：進模型前先整理輸入

官方文件說 Headroom 會處理 tool outputs、DB results、file reads、RAG results、API responses 等內容，再把較乾淨的 prompt 送往 LLM provider。它可以用 proxy、SDK、framework integration 或 MCP tools 方式接入。

最重要的判斷：不要直接裝進核心

Hermes 的核心仍應是 Obsidian/Wiki Agent、Codex skill sync、Summary Agent 與原始資料保存。Headroom 可以做輔助層，但不應成為唯一記憶、唯一證據或唯一知識來源。

整體關係圖：上下文清理器在核心外圍

Context Headroom 先做壓縮、去重、摘要、異常保留與可回取索引，只給 agent 看乾淨輸入。

閱讀方式：Headroom 類工具應放在「Raw Sources」與「Agent Decision」之間。它不取代 raw archive，也不直接改 Wiki 正式知識。
3

核心概念與名詞解釋

Context Optimization Layer
模型看到資料前的整理層。
核心概念
  • 它的任務是降低輸入雜訊，不是永久儲存知識。
  • 適合處理工具輸出、log、DB rows、RAG chunks、API response 與長 transcript。
  • 對本地 LLM 的價值是穩定長任務，而不是單純省 token 成本。
ADLINK 說法：把它當成進會議前的助理摘要，不是公司的正式文件庫。
SmartCrusher / Content Router
依資料型態選擇壓縮策略。
核心概念
  • 官方文件描述會辨識 JSON、code、logs、text、diffs、HTML，再套用不同 transform。
  • JSON array、structured logs、API rows 通常比短對話更值得壓縮。
  • 壓縮時應保留 errors、anomalies、boundary 與代表性樣本。
ADLINK 說法：Excel issue list 應保留異常、風險與 owner，不需要把每列重複欄位塞給 agent。
CCR：Compress, Cache, Retrieve
壓縮後仍可回取原文。
核心概念
  • 官方架構說被壓縮或丟棄的內容可存入 local compression cache。
  • 模型需要細節時，可透過 retrieve tool 取回原始內容或篩選結果。
  • 這個設計很適合長 transcript 與 log，但原始 raw 仍要由 Hermes 自己保存。
注意：CCR 不是長期知識庫。TTL、儲存位置、敏感資料策略都要先確認。
Proxy、SDK、MCP Tools
三種接法，風險等級不同。
導入概念
  • Proxy 會攔截 LLM traffic，導入最方便，但治理風險最高，不能先上核心。
  • SDK 適合特定 pipeline，例如 Wiki raw 轉 Evidence Notes。
  • MCP tools 適合讓 agent 按需壓縮與回取，較符合 sandbox 測試。
ADLINK 說法：先用 MCP 或離線 script 做試驗，不要一開始改全部 provider base URL。
Cache Optimization
讓 provider prefix cache 更容易命中。
效能概念
  • 官方架構描述會把動態日期、UUID、session token 等移到後段，保留穩定 prefix。
  • 對雲端 API 有成本與延遲價值，對本地 LLM 則不是首要收益。
  • 對 Hermes，穩定系統提示與乾淨交接比 cache 折扣更重要。
ADLINK 說法：本地 LLM 不需要為了 cache 折扣犧牲 prompt 可讀性。
Memory Boundary
壓縮層不能跨過知識邊界。
治理概念
  • Wiki Agent 才能把知識變成正式長期內容。
  • Summary Agent 才負責交接狀態與下次工作入口。
  • Context Headroom 只產生 agent 可讀摘要、Evidence Notes 與風險清單。
禁止：不要讓壓縮器直接刪 raw、覆蓋 Wiki、改正式知識或吞掉精確 diff。

適用與不適用：先分清楚，不要迷信工具

最適合 Michael 系統的場景

Agent 場景適合壓縮的資料輸出成什麼
Codex / coding agentterminal output、build log、test log、長 shell output錯誤摘要、失敗原因、下一步、保留關鍵 stack trace
Wiki AgentYouTube transcript、NotebookLM raw notes、長文章與重複整理稿Evidence Notes、Lesson Brief、缺資料清單
art agent教學影片逐字稿、來源研究、HTML 修改紀錄主題、章節、術語、可引用證據、審稿 checklist
ADLink_PMExcel、email、critical issue report、客訴記錄異常、風險、owner、deadline、下一步
每日維護報告agent status、log、容量、模型、重複檔案清單乾淨結論、紅黃綠狀態、需人工確認事項

不適合直接壓縮的內容

  • `.env`、token、cookie、session、API key。
  • 會被用來改程式的精確 diff、patch、SQL migration。
  • 正式財務、法律、醫療與合規內容，除非保留精確來源與可回查證據。
  • Wiki 正式知識，不應由壓縮器直接覆寫。
  • 客戶敏感資料，尤其是會送到外部服務或 telemetry 的流程。
治理原則：壓縮給 agent 看，原始資料仍留 raw。agent 需要引用、寄出或寫入 Wiki 時，必須回查原文。
5

建議導入階段：先規則，再 sandbox，最後才接工具

階段 1

自建壓縮規則

先定義 Hermes Context Headroom 規則：哪些可摘要、哪些必須保留原文、哪些要進 Evidence Notes。

階段 2

強化交接摘要

讓 Summary Agent 與 handoff summary 固定輸出完成事項、決策、風險、下一步與原始資料路徑。

階段 3

Wiki raw 轉 Evidence

把 transcript、NotebookLM 與 raw notes 先壓成 Evidence Notes，再由 Wiki Agent 審核吸收。

階段 4

Headroom sandbox

在非核心資料上測試 Headroom MCP 或 SDK，觀察壓縮率、漏失、回取能力與本機資源負載。

階段 5

穩定後再接入

若效果穩定，再只接 Wiki、Codex log、art transcript 幾個管線，不接 secrets 與正式知識寫入。

本教材結論

Headroom 值得吸收，但不要直接裝進核心。它是上下文清理器，不是大腦。Hermes 的核心仍是 Obsidian/Wiki Agent、Codex skill sync、Summary Agent 與 raw archive。

先做的實作規則

  • 所有壓縮摘要都要附 raw path 或來源 URL。
  • 寄出、寫 Wiki、改 code 前要回查原文。
  • tool output 超過一定長度才壓縮，短內容不要多此一舉。
  • 保留 error、warning、owner、deadline、決策、未解問題。

一頁式重點

本課重點整理

  • Headroom 有用：它能幫長任務降低上下文雜訊，特別是 tool output、log、RAG、file reads、API responses 與 transcript。
  • 對本地 LLM 的價值不是省錢：Michael 的系統更需要穩定長任務、乾淨交接、避免 context 爆掉。
  • 不要取代記憶庫：它不是 Obsidian、Wiki Agent、Codex skill sync 或 Summary Agent 的替代品。
  • 先自建規則：先定義壓縮邊界、raw 保留、Evidence Notes、回查流程，再測 Headroom sandbox。
  • 先試三個管線：Wiki raw 轉 Evidence Notes、Codex build/log 摘要、art transcript 轉教學 brief。
  • 安全邊界要硬：不要壓縮 secrets、精確 diff、正式合規內容，不要讓它直接改 Wiki 正式知識。
一句話：Headroom 是上下文清理器，不是大腦。它可以讓 agent 看得更清楚，但不能替代 Hermes 的知識治理與 raw archive。
