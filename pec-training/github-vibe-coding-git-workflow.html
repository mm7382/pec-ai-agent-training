Git / GitHub Safety Workflow

先會存檔，AI 才能放心加速

Claude、Cursor、Codex 問你 commit、branch、PR，不是在講艱深指令，而是在問：要不要存檔、隔離風險、送審與回復。

不用背指令，但要會下判斷。
Commit 先建立安全檢查點
Branch 把新功能隔離開
Worktree 多個 AI 各自工作
PR 送出變更提案
Merge 審核後合回主線
Git本機版本控制，負責記錄與回復。
GitHub雲端協作平台，負責備份、分享、審查。
AI Agent寫程式很快，但需要 Git 做安全邊界。

先分清楚兩件事

Git 和 GitHub 經常被混在一起。對 AI coding 來說，這個差異會影響你要不要讓 agent 上傳、回復或合併。

Local Control

Git 是本機版本控制

它負責追蹤資料夾裡的每次改動。當 AI 改壞程式時，你可以知道改了什麼，也可以回到上一個正常狀態。

Cloud Collaboration

GitHub 是雲端協作平台

它負責保存 repo、分享給同事、發 PR、審查與合併。push 之前一定要先確認不要把 key、密碼或私密檔案送上去。

十個術語的工作判斷

不要先背命令。先理解每個詞在 AI 開發流程裡到底解決什麼風險。

Checkpoint

commit

把目前可用狀態存成一個檢查點。AI 問你要不要 commit，就是問你要不要先買保險。

Upload

push

把本機 commit 推到 GitHub。push 前先檢查 `.gitignore`，不要讓 API key 或公司私密檔案上雲端。

Join

clone

正式把 repo 複製到本機，保留 Git 歷史與同步能力。它不是單純 Download ZIP。

Sync

pull

把 GitHub 上其他人或其他 agent 的最新進度抓回本機。PR merge 後，本機 main 也要 pull。

Sandbox

branch

開一條不影響 main 的實驗線。新功能、大修改、AI 不確定的任務，都應該先在 branch 做。

Parallel

worktree

幫不同 branch 建立不同實體資料夾。多個 AI agent 同時工作時，用 worktree 避免互相踩檔案。

Review

PR

Pull Request 是變更提案。讓同事或 reviewer agent 先看改了什麼、怎麼測，再決定要不要合併。

Approve

merge

把審核後的 branch 合回 main。merge 在 GitHub 完成後，本機 main 還要 pull 才會同步。

Undo

restore

還沒 commit 的錯誤修改，可以恢復到上一個正常檢查點。適合「AI 剛改壞，我不要這些改動」。

Safe Undo

revert

已經 commit 的錯誤，用反向 commit 取消。協作時比偷偷改歷史安全，大家看得到發生過什麼。

多個 AI 同時做事時怎麼不打架

branch 像同一張桌子的不同狀態；worktree 則是多張實體桌子。AI agent 並行時，要給它們不同工作區。

只用 branch

A
資料庫功能 branch可以隔離 main，但同一個資料夾一次只能呈現一個 branch。
B
UI 重設計 branch如果兩個 agent 在同一資料夾切來切去，容易混入錯誤狀態。
改用
worktree

branch + worktree

A
Agent A 專用資料夾只處理資料庫功能，commit 到自己的 branch。
B
Agent B 專用資料夾只處理 UI 重設計，互不污染，再各自開 PR。

AI Coding 的五個安全閘門

讓 AI 寫得快之前，先把哪些事情一定要檢查清楚。這些是降低災難成本的基本規則。

push 前查機密

要求 agent 確認 `.env`、API key、token、密碼都已在 `.gitignore` 或未被追蹤。

大改前先 commit

目前能跑的版本先存檔，AI 改壞時才有明確回復點。

新功能開 branch

main 保持穩定，新功能、重構、UI 大改都放到獨立 branch。

merge 前開 PR

PR 要說明改了什麼、測了什麼、哪些檔案受影響，方便人或 reviewer agent 檢查。

合併後 pull

GitHub 上 merge 完，本機 main 不會自動更新。下一個 branch 前先 pull。

你可以直接這樣交辦 AI

重點是把風險邊界說清楚。你不需要背 Git 指令，但要能描述你要的狀態。

開始新功能

請先從 main 開一個新 branch 做這個功能，所有修改都 commit 在這個 branch，不要直接動 main。

push 前檢查

push 前請檢查 `.gitignore`，確認 `.env`、API key、token、密碼與公司私密資料沒有被 commit。

多 agent 並行

請為這個任務建立獨立 worktree，讓它和目前工作資料夾分開，避免兩個 agent 改到同一批檔案。

準備 PR

請為這個 branch 建立 PR，說明完成內容、修改檔案、測試方式、風險點與 reviewer 要注意的地方。

處理 conflict

請分析 conflict，保留朋友的新邏輯，但把我新增的分類移到右側進階選單，再跑測試確認。

改壞回復

我不要剛剛那批錯誤修改。請判斷目前狀態，若尚未 commit 就 restore，若已 commit 就用安全方式 revert。
