Report Brief

這堂課要解決的問題

  • 多數團隊已經同時使用 Claude、ChatGPT、Gemini、Codex、本地模型與 Telegram agent，但工具之間缺少統一狀態、任務紀錄、成本視圖與改善回饋。
  • 真正的瓶頸不是單一 prompt 品質，而是 AI 工作流沒有像系統一樣被觀測、管理與持續改進：誰在使用、花多少 token、產生什麼價值、哪些知識已過期、哪些 skills 已失效。
  • 本課的核心是建立 Claude Agentic OS：把模型訂閱、知識庫、skills、agent 活動與自動化改善流程集中到一個可視化控制台，讓 AI 從單點工具升級為可營運的平台。
  • 課後輸出不應是逐字稿，而是一份能對內報告的系統設計教材：包含架構、導入步驟、資料模型、KPI、治理風險與落地情境。

教學對象

  • AI Agent / Hermes / Codex / Claude Code 使用者：已經有多個 agent、skills 或工作流，需要集中管理與量化成效的人。
  • 系統工程師與平台負責人：需要把 AI 工具納入可觀測性、權限、資料來源、排程、成本與風險控制的人。
  • PM / 技術主管：需要知道 AI 投資是否有產出、哪些專案使用頻率最高、哪些知識庫需要維護、哪些自動化值得擴大導入的人。
  • 內部工具開發者：想把 agent 能力包成可交付 dashboard，讓非工程角色也能看懂 AI 工作流狀態的人。
  • 不適合對象：只想看單一聊天工具示範、尚未累積任何 agent / skills / 知識庫，也不需要管理成本與流程的人。
Visual Roadmap

Claude Agentic OS 架構

01Layer 1

AI Usage Dashboard - 目的為建立成本與額度觀測；資料來源包含訂閱方案、token usage、模型類型與任務類型；輸出為月成本、額度壓力、模型選型建議。

02Layer 2

Knowledge Hub - 目的為統一知識來源與新鮮度；資料來源包含 Obsidian、Pinecone、Notion、HTML 教材、本地記憶與專案文件；輸出為關聯圖、過期文件、缺少索引的資料。

03Layer 3

Skills Registry - 目的為管理可重複能力；資料來源包含 Codex skills、Hermes tools、agent prompt、風格規範與腳本；輸出為使用頻率、成功率、冷門技能、可封存清單。

04Layer 4

Agent Activity Map - 目的為追蹤 agent 實際產出；資料來源包含 Hermes Agent、Claude Code、Codex、Telegram bot、local LLM 任務紀錄；輸出為專案活動、任務類型、產物連結與 owner。

05Layer 5

Dreaming Engine - 目的為自動產生改善提案；資料來源為前四層觀測資料；輸出為成本優化、知識補強、skills 重訓、專案規則修正與人工審核清單。

System Workflow

工程師操作模型

STEP 1Step 1

定義導入範圍 - 先決定要管理個人工作站、單一專案，或整個 agent 團隊；同步定義 KPI，例如月成本、節省工時、知識過期率與建議採納率。

STEP 2Step 2

安裝 OS template - 放到 VS Code、Cursor 或既有專案資料夾，讓 Claude 依 `CLAUDE.md` 或專案指令建立 dashboard、本機服務與資料目錄。

STEP 3Step 3

建立工具清冊 - 掃描 Claude App、Claude Code、Codex、Gemini CLI、Telegram bot、本地 LLM、VS Code extension 與自動化腳本，標記 owner 與用途。

STEP 4Step 4

連接知識來源 - 串接 Obsidian、Pinecone、Notion、HTML 教材、會議紀錄與本機文件，並記錄更新時間、來源可信度與可檢索狀態。

STEP 5Step 5

建立價值模型 - 設定人員角色、時薪或每小時價值、任務類型與節省時間估算方式，讓每個 AI artifact 都能回推成本效益。

STEP 6Step 6

設定 Dreaming 排程 - 每日或每週掃描前四層資料，輸出改善建議；所有建議都必須包含原因、影響範圍、預期收益與 owner review 狀態。

Engineering Applications
1

成本雷達

指標包含月成本、token 使用量、模型比例與任務分類；動作是把低風險任務移到較低成本模型，並保留高推理任務給高階模型。

2

技能盤點

指標包含 7/30/90 天使用次數、成功率、重複功能與最後修改日；動作是封存冷門 skills、合併重複 prompt、安排需要重訓的風格規範。

3

專案健康檢查

指標包含是否有 `CLAUDE.md`、是否有測試指令、是否有產物目錄、是否有 owner；動作是補 onboarding、補驗收標準與補常見任務規則。

4

知識庫視覺化

指標包含來源數量、更新時間、孤立節點、重複文件與引用次數；動作是補索引、標記過期內容、把教材與任務紀錄連回來源。

5

價值計算

指標包含節省時間、人工時薪、重工率與交付件數量；動作是把 AI 產出轉成可量化的節省金額與產能提升。

Teaching Notes

核心觀念

  • 從 AI 工具孤島升級到 AI Operating Layer：重點不是再安裝一個工具，而是把既有工具接到同一套可觀測與治理框架。
  • Dashboard 不是裝飾介面，而是 AI 工作流的 observability layer；它要能回答成本、價值、資料狀態、agent 活動與改善優先順序。
  • Dreaming 不是完全自動決策，而是排程式改善迴圈：系統先提出建議，再由 owner review，最後把採納結果回寫成下一輪資料。
  • Agentic OS 的成熟度來自資料品質：如果 token、任務紀錄、知識庫版本、skills 使用率與產物連結不完整，dashboard 只會變成漂亮但無法決策的頁面。
  • 工程師導入時要把每個 AI 輸出視為 artifact：HTML 教材、prompt、腳本、會議摘要、規格整理、測試報告都應該有來源、版本、owner 與驗收狀態。

資料模型與指標

  • ModelSubscription：記錄工具名稱、方案、額度、月費、主要模型、token 使用與續約狀態，用於成本治理。
  • AgentProfile：記錄 agent 名稱、入口介面、任務範圍、owner、使用模型、可用工具與最近產出，用於 agent portfolio 管理。
  • SkillDefinition：記錄 skill 名稱、觸發條件、依賴工具、成功案例、最後訓練日期與封存狀態，用於 skills lifecycle 管理。
  • KnowledgeSource：記錄文件類型、來源路徑、更新時間、可信度、索引狀態與引用次數，用於知識庫新鮮度與追溯。
  • ProjectActivity：記錄專案、任務、輸入來源、輸出 artifact、執行時間、人工確認結果與可重複性，用於衡量 AI 導入價值。
  • DreamingRecommendation：記錄建議內容、觸發原因、影響範圍、預期收益、風險、owner review 與採納結果，用於建立持續改善閉環。
  • 核心 KPI：token 成本、訂閱利用率、agent 使用頻率、skill 成功率、過期知識比例、孤立知識節點、節省工時、建議採納率、回歸錯誤率。

ADLINK 可落地場景

  • Agent 管理中心：集中查看小愛、art_bot、PM bot、Telegram bot、Codex 任務與本地 LLM 工作流，讓每個 agent 都有定位、owner、產物與使用狀態。
  • 教學教材產線：art_bot 接收影片或文件後，自動取得內容來源、轉成 ADLINK 內訓 HTML、提供預覽、等待檔名與儲存位置確認，再存到 Mac mini 本機。
  • PM / 工程協作看板：把需求整理、會議紀錄、規格摘要、客戶問題與測試紀錄連到同一個任務流，降低資訊散落在聊天與本機檔案中的風險。
  • 成本與效益看板：追蹤不同模型、agent、任務類型的 token 成本與節省工時，找出高成本低價值任務，並提出模型切換建議。
  • 知識庫治理：檢查內訓 HTML、產品資料、會議紀錄、PM 文件是否過期、重複、缺少索引或沒有來源，避免 agent 用舊資料回答。
  • 技能維護：找出長期不用的 skills、重複功能、失效依賴與需要重訓的 ADLINK 風格規範，讓 agent 能力庫保持乾淨。
  • 夜間自動改善：每天或每週產出改善建議，包含重構腳本、封存技能、補文件、更新知識庫、降低成本與補驗收流程。

導入風險與控制

  • 權限風險：API key、Telegram token、Notion token 與本地檔案權限必須分層保存；dashboard 只能顯示狀態，不應直接暴露 token 原文。
  • 資料風險：知識庫來源要有更新時間、來源路徑與可信度；沒有來源或無法追溯的內容只能標成 draft，不能進入正式教材。
  • 自動化風險：Dreaming 建議不得直接修改正式資料；要先進入 review queue，由 owner 確認後才可套用、封存或刪除。
  • 成本風險：模型切換不能只看便宜；需要依任務風險分級，將高推理、客戶輸出與工程判斷保留在高可靠模型。
  • 品質風險：每個 HTML 教材至少要檢查繁體中文、來源一致性、內容是否具體、是否出現空洞章節、是否誤把逐字稿當教案。
  • 營運風險：agent 數量增加後必須有 owner、版本、用途、停用條件與 rollback 設計，否則控制台會變成另一個資訊孤島。

常見誤解

  • 誤解 1：Agentic OS 是另一個聊天工具。實際上它是管理多個 AI 工具與工作流的控制台，負責觀測、治理與改善。
  • 誤解 2：只要有 Claude Code 就夠。課程重點是把 Claude Code、Codex、知識庫、skills、成本資料與 agent 紀錄放到同一個系統中。
  • 誤解 3：Dashboard 只是漂亮介面。真正價值在於看見成本、使用率、過期知識、閒置技能、專案健康度與可改善流程。
  • 誤解 4：Dreaming 是自動魔法。它仍需要清楚的資料來源、工作區規則、owner review 與採納紀錄，否則只會產生大量無法執行的建議。
  • 誤解 5：導入一定要一次做完整平台。更務實的方式是先選一個高價值流程，例如教學 HTML 產線或 agent 管理中心，做出可驗收的最小版本。

工作坊設計

  • 0-10 min：盤點目前團隊使用的 AI 工具、agent、知識庫與自動化腳本，先列出工具清冊與 owner。
  • 10-25 min：畫出現有 AI 孤島，標記哪些工具沒有共享狀態、成本、任務紀錄、產物連結或知識來源。
  • 25-40 min：設計第一版 Agentic OS dashboard，至少包含成本、skills、知識庫、agent activity 與 Dreaming review queue。
  • 40-55 min：選一個真實工作區建立 `CLAUDE.md` 或 agent instruction，定義可掃描的資料目錄、測試指令、輸出格式與驗收標準。
  • 55-75 min：建立第一版資料模型，填入 3 個 agent、5 個 skills、5 個 knowledge sources 與 3 個 project activities。
  • 75-90 min：產生第一輪 Dreaming 建議，分成採納、拒絕、延後與需要人工確認，並定義下週要追蹤的 KPI。

驗收清單

  • 能說明 AI 工具孤島會造成的管理問題：成本不可見、知識不可追溯、skills 失效、agent 產物散落與改善建議無法閉環。
  • 能畫出 Agentic OS 的五層架構：Usage、Knowledge、Skills、Agents、Dreaming，並說明每層的資料來源、輸出與 owner。
  • 能列出 dashboard 必須追蹤的指標：token、成本、訂閱利用率、agent 活動、skill 成功率、知識庫新鮮度、節省工時、建議採納率。
  • 能設計一個 ADLINK 內部 agent 管理中心的最小可行版本，至少包含 art_bot、小愛、PM bot、Codex 任務與本地 LLM 工作流。
  • 能為 HTML 教材產線定義端到端流程：來源取得、內容理解、ADLINK 風格生成、預覽確認、檔名確認、儲存位置確認與本機歸檔。
  • 能判斷哪些改善建議可自動採納，哪些必須由 owner review，並保留 audit log 與 rollback 路徑。
  • 能辨識不合格輸出：內容空洞、與來源不相關、出現簡體字、只貼逐字稿、缺少工程應用、沒有來源或沒有驗收標準。
