Public Learning Note

Codex 插件與 Skill 入門：6 種工作流的導入判斷

把 plugin 當成「可安裝能力」，把 skill 當成「可重複流程」。重點不是裝越多越好，而是先看任務類型，再決定要用 Remotion、UI/UX、File Organizer、Deep Research、Data Analytics 或 Record & Replay。

一句話結論

先用小任務驗證，再把穩定流程寫成 skill；能碰資料、瀏覽器、檔案或表單的 plugin，一律先 sandbox 測試，不直接放進核心。

任務要接外部工具
優先看 plugin 流程會重複執行
優先寫 skill 固定操作要示範
再考慮錄製

先分清楚：Plugin、Skill、Record & Replay

Mental Model
P

Plugin：能力包

可把工具、MCP、connector、角色型工作流一起包起來。適合需要連外部系統、資料源、瀏覽器、設計工具或分析工具的任務。

S

Skill：流程包

把固定做法、檔案範本、規則與腳本寫成可重複使用的工作流。適合 SOP、報告格式、檔案整理、研究流程。

R

Record & Replay：示範轉流程

把一次操作示範轉成 reusable skill。適合固定表單、固定網站流程、重複文件處理；不適合錄入密碼或敏感資料。

Decision Rule

六種能力怎麼用

1. Remotion：資料與流程動畫

適合把數據排名、流程變化、產品故事做成可反覆修改的動畫。

Remotion 的價值在於「動畫是程式碼」。第一版不漂亮沒關係，後續可以請 Codex 修改節奏、轉場、顏色、圖表比例與文字層級。

適合銷量排行、技術演進、產品比較、維護流程、展示型短片。
注意資料來源要先確認；正式對外素材仍要人工審稿。
React動畫資料視覺化

2. UI/UX Pro Max：教學頁與看板美化

適合把文字整理成 HTML、dashboard、內部教學頁或可讀的工作台。

這類 skill 不是替你決定內容真偽，而是協助排版、資訊層級、配色、元件模式與互動細節。對內容輸出 agent 來說，應放在「內容已整理好」之後使用。

適合Michael 教學筆記 HTML、客戶問題整理頁、PM 報告看板。
注意第三方 skill 要先讀 README、license、安裝腳本，不要盲裝。
HTML報告Dashboard

3. File Organizer：資料夾整理與重新命名

適合清理 Downloads、素材庫、截圖資料夾、專案暫存檔。

好的整理流程不應一開始就搬檔，而是先產生分類計畫、重命名規則與衝突清單。確認後再執行，且要保留回復方式。

適合把圖片、PDF、報告、學習素材依主題與日期整理。
注意公司資料、客戶資料、合約資料不可讓未審查 skill 自動搬移。
Dry-run備份命名規則

4. Deep Research：市場與技術研究

適合市場概況、競品比較、文獻整理、採購前評估。

研究型 skill 的重點不是字數，而是可追溯：每個結論要有來源，沒有證據時要明確寫「目前未找到可靠證據」。

適合台股報告、AI 工具調研、產業趨勢、Hermes 能力雷達。
注意財務、法律、醫療等高風險內容要保留原始來源並再次查證。
Evidence比較表不確定性

5. Data Analytics：資料分析與決策報告

適合把 Excel、CSV、營運資料變成指標、洞察與視覺化報告。

可以用來做資料清理、轉換率分析、商品分級、異常偵測、趨勢圖與行動建議。對 PM workflow 場景特別適合處理 Excel 與 issue report。

適合GMV、轉換率、維修統計、客訴分類、專案風險摘要。
注意不能只看漂亮圖表；要附上欄位定義、計算方式、資料缺口。
ExcelCSV指標報告

6. Record & Replay：把固定操作變成 skill

適合固定、可示範、低變動的瀏覽器或桌面流程。

它的價值是讓 Codex 看一次標準操作，再把步驟、輸入、成功條件整理成可重用 skill。正式使用前一定要先用測試資料跑過。

適合固定表單、固定資料下載、固定命名、固定報表產出。
注意不要錄密碼、token、cookie、客戶機密或不可外流畫面。
重複任務Mac權限控管

導入 SOP：不要一看到就裝

1

先定義任務

輸入是檔案、資料表、網頁、圖片，還是固定操作？輸出是報告、動畫、整理後資料夾，還是可重複 SOP？

2

查來源與權限

看官方文件、GitHub、README、license、安裝腳本、需要的 connector 或電腦控制權限。

3

用測試資料跑

先用 sample file、dummy folder、假表單資料測試。不要一開始就接 正式資料。

4

留下驗收標準

確認輸出是否可讀、可追溯、可回復；資料分析要有計算方式，檔案整理要有搬移清單。

5

再寫成 Hermes 規則

穩定後才放進 Codex skill sync 或 Wiki Agent。只吸收流程，不吸收 token、raw chat、暫存資料。

風險與安全邊界

Do
  • 先用 sandbox 專案測試 plugin / skill。
  • 把來源、日期、版本與安裝方式記錄下來。
  • 要求 Codex 先列出會讀寫哪些檔案、會連哪些工具。
  • 對資料分析報告要求欄位定義、公式、資料缺口。
  • 對 Record & Replay 只錄固定步驟，不錄機密輸入。
Don't
  • 不要把未審查第三方 plugin 放進 Hermes 核心。
  • 不要讓整理工具直接搬移公司正式資料夾。
  • 不要把 `.env`、token、cookie、密碼、客戶資料放進 skill。
  • 不要把研究型 skill 的輸出直接當 Wiki 正式知識。
  • 不要用漂亮看板取代來源查證與人工審核。
Michael 的建議導入順序 先把 UI/UX、Data Analytics、Deep Research 當成「任務輔助」；File Organizer 與 Record & Replay 因為會碰檔案與電腦操作，放在第二階段，通過 dry-run 與權限檢查後再導入。

可直接複製的提示詞

判斷要不要用 plugin / skill

請先判斷這個任務適合：
1. 一般 Codex 對話
2. plugin
3. skill
4. Record & Replay

請列出理由、需要的權限、可能風險、驗收方式。
在我確認前，不要安裝、不要搬檔、不要連接正式資料。

資料分析報告

請用 Data Analytics 思路分析這份資料。
先說明欄位定義與資料缺口，再整理：
- 主要 KPI
- 異常或風險
- 可能原因
- 建議行動
- 需要人工確認的地方
最後輸出成 個人教學報告格式。

檔案整理

請先掃描這個資料夾並提出整理計畫。
只輸出 dry-run，不要直接移動或改名。
請列出：
- 建議分類
- 檔名規則
- 重複檔
- 不確定檔案
- 執行後可回復方式

把流程寫成 skill

請把這次已驗證可用的流程整理成 skill 草案。
只保留穩定 SOP、輸入輸出、限制、驗收方式。
不要寫入 token、cookie、個人信箱密碼、暫存路徑或原始聊天內容。
