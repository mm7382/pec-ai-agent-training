Agent Decision Control

Grill Me：先把決策問清楚，再讓 Agent 動手

真正有價值的不是「AI 問很多問題」，而是把事實查證、選項建議、決策依賴與人類確認，組成一個不會偷跑的開工閘門。

本機稽核結論 已重新依 Matt Pocock GitHub `main` 的 2026-07-21 commit 核對：官方核心 portfolio 已完整安裝並對齊。Codex 也已加入繁體中文自然語句路由；目前剩下 invocation log 與 smoke test 證據需要補強。
真正結構 五項規則 實際對話 完整工作流 Skill Portfolio 架構思維 本機稽核 差異結論 提示詞 Sources

02 · Operating Rules

五項規則少一項，就不算真的 Grill Me

判斷 agent 有沒有照做，不是看它問了幾題，而是看問題的順序、資訊來源、建議品質與停止條件。

RULE 01

一次只問一題

每次處理最重要的未決項目，等回答後才走下一個分支。

檢查點：不能一次丟十題問卷。

RULE 02

沿決策樹往下

先解上游選擇，再問受它影響的下游問題，避免順序顛倒。

檢查點：每題應能說明為何現在要問。

RULE 03

每題附建議答案

Agent 不只把思考工作丟回來，還要提供推薦選項、理由與取捨。

檢查點：有 recommendation，不是空泛追問。

RULE 04

事實自己查

檔案位置、現有行為、API、程式流程等可查事實，由 agent 探索環境取得。

檢查點：只把真正需要人類拍板的 decision 問你。

RULE 05

確認前不動工

達成 shared understanding 並由人類確認前，不寫 code、不改設定、不開始實作。

檢查點：有明確 approval gate。

同一句需求，直接做與先釐清的結果差很多

以下以「讓內容輸出 agent 自動交付最新教學檔」為例。真正重要的不是多問，而是先抓出會改變安全性與使用方式的上游決策。

  • 收件人是否固定 指定收件信箱？
  • 寄送失敗如何顯示與重試？

04 · Idea To Delivery

Grill Me 只是入口，後面要接可驗證的工程流程

每個 skill 只完成一個明確工作。需求已經清楚時可以從中間開始，不需要為了遵守形式把整條線重跑。

01 grilling

把未決問題、限制、取捨與人類決策問清楚。

02 to-spec

把共識整理成問題、方案、用戶故事、決策、測試與邊界。

03 to-tickets

拆成可獨立驗證的 tracer-bullet tickets，標出 blocking edges。

04 implement

只依已確認 spec / ticket 實作，不重新發明需求。

05 TDD

在預先同意的 seam 上，一次完成一個 red → green 垂直切片。

06 code-review

分開檢查 Standards 與 Spec，避免「寫得漂亮但做錯需求」。

正確分流：沒有 codebase 的一般想法用 `grill-me`；已經在專案裡做功能、架構或領域設計，使用 `grill-with-docs`。後者會把 canonical terms 寫入 `CONTEXT.md`，只有難以逆轉、令人意外且存在真實取捨的決策才值得 ADR。

05 · Current Skill Portfolio

不是所有 skill 都自動跑：Matt 刻意分成「流程入口」與「工程紀律」

這是前一版最需要說清楚的地方。官方 `main` 以 `disable-model-invocation` 區分誰能啟動：手動入口控制階段，自動 skill 只在任務符合時提供可重用紀律。全部一起觸發反而會增加 random 行為。

手動入口

grill-with-docs

你口述的「grow with stacks」最可能是這個。用於已有 codebase 的需求釐清，同時維護 `CONTEXT.md` 與必要 ADR。

手動入口

to-spec

把已經談清楚的對話整理成 spec。它不應重新訪談，也不應直接實作。

手動入口

to-tickets

你說的「to check it」正確名稱是 `to-tickets`。它把 spec 拆成可獨立驗證的 tracer-bullet tickets，並標示 blocking edges。

手動入口

implement

只在 spec 或 tickets 已核准後執行；內部使用 TDD，完成後再跑 code-review，最後才 commit。

可自動採用

tdd

對可測行為執行 red → green → refactor。測公開行為，不綁死內部實作。

可自動採用

code-review

從固定基準點比較 diff，分開檢查 Standards 與 Spec，避免程式碼品質掩蓋需求偏差。

可自動採用

diagnosing-bugs

先建立會穩定變紅的重現迴圈，再最小化、假設、觀測、修復與回歸測試；不靠猜測直接改。

手動入口

improve-codebase-architecture

掃描 deep-module 改善機會並產生視覺報告。先選候選項，再回到 grill-with-docs，不會自行重構。

手動入口

wayfinder

只給超過單一 context window 的大型模糊工作。它解決 decision tickets，不產生功能；路線清楚後才接 to-spec。

本機狀態：官方 README 現列的 22 個工程與生產力核心 skill 已全部存在於 Codex。`to-tickets` 唯一多出的舊句子也已移除，現在與官方 commit `ed37663` 完全一致；不需要重裝整套或改動 upstream frontmatter。

06 · Deep Module Thinking

好架構不是檔案越多，而是介面越小、內部能力越深

AI 維護 codebase 時，最怕為了理解一個行為在許多薄檔案間跳轉。深模組把複雜性藏在小介面後面，讓修改、測試與知識集中在同一個地方。

Shallow module

介面接近實作一樣複雜，呼叫者必須知道太多細節。

大型介面：很多方法、參數與順序規則
薄實作：多半只是轉接或傳遞

Deep module

用小介面提供大量行為，複雜性、測試與修正集中在模組內。

小介面：少量清楚操作
深實作：規則、錯誤處理、資料轉換與依賴細節都藏在裡面
Leverage + Locality

Deletion test

想像刪掉這個模組。如果複雜性散回許多 caller，它有隱藏真正複雜度；如果程式反而更簡單，它可能只是 middle man。

Interface = test surface

Caller 與測試應跨越同一個 seam。需要繞過介面才能測試，通常代表模組形狀不對。

One adapter is hypothetical

只有一個 adapter 時，不要急著建立抽象 seam；真的有兩個變體，這條 seam 才有實際價值。

08 · Differences And Decision

真正的差異不在 prompt，而在觸發、語言與治理

不需要再安裝一份 Matt skills。現在應把力氣放在「正確階段才啟動、能否證明啟動、同步時不覆蓋本機治理規則」三件事。

UPSTREAM

小型、可組合

手動 wrapper 與模型可重用 primitive 分離。使用者記得何時叫 wrapper；skill 本體保持最小。

優點：少 context load、單一真實來源、容易組合。

MICHAEL LOCAL

增加治理分流

全域 `AGENTS.md` 已讓 codebase 與非 codebase 任務走不同入口，並加入常用中文口述別名與 implementation gate。

優點：符合 Michael 不希望 agent 亂做的使用習慣。

GAP

缺少可觀測閉環

仍缺直接 invocation log 與固定 smoke test；舊 clone 也可能成為錯誤同步來源。

風險：看起來有能力，實際何時觸發卻無法證明。

09 · Ready To Use

三種情境，使用三種入口

不需要背完整 skill catalog。先判斷有沒有 codebase、是否需要留下術語與決策，再選入口。

一般想法 / 非程式任務 
/grill-me

請使用繁體中文。一次只問一個最重要的決策問題；每題附上你的建議答案與取捨。可從現有資料查到的事實請先查，不要問我。直到我明確確認 shared understanding 前，不要開始執行。
Codebase / 架構 / 領域設計 
/grill-with-docs

請先讀取現有 CONTEXT.md、ADR 與程式碼，再用繁體中文逐題訪談。每題提供建議答案。確認的新術語即時更新 glossary；只有難以逆轉、令人意外且存在真實取捨的決策才提議 ADR。未經我確認，不要實作。
檢查 agent 是否真的照規則 
請只做 Grill Me smoke test，不實作需求。

輸出並記錄：
1. 是否一次只問一題
2. 是否每題附建議與取捨
3. 是否先查可查的事實
4. 是否把真正決策留給我
5. 是否在我確認前停止執行

任何一項失敗，都要標記 fail 與原因。
