Matt Pocock Skills：Codex 需求釐清到 Ticket 工作法

這套方法的重點不是「裝更多工具」，而是讓 Codex 在寫 code 前先問清楚、寫成規格、拆成可驗證 ticket，再照 ticket 實作與 review。

一句話結論

適合放在 Codex / Codebase / Codeless 討論流程；不適合塞進每個本地 agent 當共同能力。

先釐清
再寫規格 先拆票
再動手 有驗證
才算完成
為什麼需要標準流程Skill 分工導入判斷ADLINK 用法提示詞Sources

為什麼這套方法有用

真正要解決的問題很多 AI 寫碼失敗不是模型太笨，而是需求還沒清楚、測試沒有定義、工作切太大。Matt Pocock Skills 把「想法到交付」拆成固定流程，讓 agent 不會一上來就亂改。
1

避免做錯方向

先問清楚再動手。

用 grilling / grill-with-docs 釐清目標、限制、使用者、不能做的事。

2

避免大包任務

拆成可驗證 ticket。

用 to-spec 與 to-tickets 把模糊工作變成小切片，每一片都能驗收。

3

避免改完沒證據

測試與 review 收尾。

用 tdd、diagnosing-bugs、code-review 建立回饋迴路，不只看 demo。

Codex 建議標準流程

1

ask-matt

先判斷要走哪條流程，不確定時不要直接做。

2

grill-with-docs

需求模糊時先追問，並留下術語與決策紀錄。

3

to-spec

把共識整理成規格：問題、方案、故事、決策、測試、邊界。

4

to-tickets

把規格拆成最小可驗證 ticket，標出依賴關係。

5

implement

照 ticket 實作，不重新發明需求。

6

code-review

從規格符合度與工程品質兩軸檢查。

哪些不要直接導入

不放 local agent

這套主要是 codebase 討論與實作治理，不是每個本地 agent 都需要。小愛、Wiki Agent、art agent 只需要知道「正式知識/教學/輸出」怎麼接收，不需要自己跑整套 coding 流程。

不導入 deprecated

舊版 `to-prd`、`to-issues`、`qa`、`request-refactor-plan` 這類已被新版替代，不要再擴散。以 `to-spec`、`to-tickets`、`code-review` 為準。

先不導入作者個人用途

`obsidian-vault`、`edit-article` 這類偏作者個人環境，除非 ADLINK 真的要建立同樣工作流，不然先不要放進核心。

in-progress 先觀察

`wizard`、`to-questionnaire`、`loop-me` 等還在實驗階段。可以學概念，但不要當正式 SOP。

ADLINK 實際用法

改 code 前

先釐清規格

你可以要求 Codex：先用 ask-matt 判斷流程，需求不清楚就進 grill-with-docs，不能直接改。

任務變大時

拆 ticket

一個任務超過一個檔案或多個行為時，先 to-spec，再 to-tickets，一張 ticket 做完再做下一張。

做完後

review 與交接

改完要跑測試，接著用 code-review 對照 spec；上下文快滿時用 handoff 留交接。

可直接使用的提示詞

需求不清楚時

請先用 ask-matt 判斷這個任務該走哪個流程。
如果需求還不清楚，請進入 grill-with-docs：
一次只問一個最重要問題，
每題給你的建議答案與取捨，
等我確認後再寫 spec，不要直接實作。

準備實作時

請把目前共識整理成 to-spec 格式：
- 問題描述
- 解決方案
- 用戶故事
- 實作決策
- 測試決策
- 邊界條件
如果工作超過一個小修改，請接著拆成 to-tickets。
