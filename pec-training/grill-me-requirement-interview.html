AI Agent 開工前需求訪談：Grill Me 工作法

AI 很認真亂做，通常不是因為模型太弱，而是需求還沒被問清楚。Grill Me 的價值，是讓 agent 在動手前先把受眾、限制、輸出、技術選擇與不能做的事全部釐清。

一句話結論

不要一開始就叫 agent 做。先讓它「拷問需求」，把模糊想法變成可執行 brief，再交給 PRD、TDD 或實作流程。

Grill Me 是什麼：開工前的需求檢查官

Workflow

Grill Me 不是魔法 prompt，也不是自動化工具。它是一種「先訪談再執行」的工作法：當需求還只是方向，agent 必須先追問你，直到任務能被清楚執行。

1

模糊想法

例如：做一個側邊欄小遊戲、做一份教學檔、整理一個客戶問題。

2

反向訪談

agent 先問使用情境、限制、成功標準與不能做的事。

3

需求定稿

把回答整理成 brief、PRD 或任務規格。

4

測試設計

用 TDD 或驗收清單定義紅燈與綠燈。

5

開始執行

交給 Codex、Claude Code、PM workflow agent 或內容輸出 agent。

Grill Me 應該問哪些問題

1

任務邊界

到底要做什麼、不做什麼。

確認目標、範圍、一次性原型或長期工具、是否需要保存資料、是否需要可重複使用。

2

使用情境

誰在什麼環境使用。

釐清是側邊欄、桌面、網頁、手機、團隊工具或客戶場景；不同入口會影響操作設計。

3

互動與限制

怎麼操作、哪些不能碰。

確認鍵盤、滑鼠、觸控、通知、權限、資料保存、清除紀錄，以及安全與隱私限制。

4

成功標準

什麼叫做完成。

用驗收條件描述成果：能否正常操作、資料是否正確、畫面是否符合使用者情境。

5

測試方法

怎麼知道沒有壞。

把需求轉成紅燈/綠燈測試、手動驗收或 Playwright 檢查，避免只看 demo 感覺。

6

後續維護

做完之後誰接手。

確認檔案位置、文件、使用說明、錯誤處理、重設方式，以及是否需要寫成 skill。

差異：直接做 vs 先訪談

面向直接叫 agent 做先用 Grill Me 訪談實務判斷
需求品質agent 會自行猜測未定義細節。未定義處會被先問出來。客戶/內部專案應先訪談。
功能完整度可能能跑，但缺模式、紀錄、通知、重設等情境細節。功能來自明確使用情境，而不是隨機腦補。正式工具要有情境設計。
返工風險後面才發現方向錯，常要重構。早期先對齊，降低後續大改。越高風險越要先問。
測試方式多半只看 demo 能不能跑。可轉成 PRD 與 TDD 驗收。Codex 實作前要有測試標準。
適用任務小型一次性草稿。專案、報告、工具、agent workflow。Hermes 穩定流程都該採用。

套到 Hermes / Agent workflow 的用法

教材產製

教學檔開工前

先問讀者是誰、看完要會什麼、需要哪些圖解、哪些字眼不能出現、是否要手機預覽與分享格式。

PM workflow

客戶/issue 報告前

先問異常定義、影響範圍、owner、deadline、證據、下一步，以及哪些資料不能外流。

Codex

改 code 前

先問檔案範圍、預期行為、不可改動區域、測試方法、rollback 條件，避免一改就散。

Wiki Agent

研究前

先問研究問題、可信來源、時間範圍、信心等級、是否需要進 raw/inbox，而不是直接寫成正式知識。

安裝與導入安全

不要直接叫 agent 安裝整包未審查技能第三方 skill 可能包含腳本、外部依賴或不符合 個人或團隊資料規範的流程。先讀 SKILL.md、README、license、安裝位置與會讀寫的檔案，再用 sandbox 專案測試。
建議做法
  • 只先安裝單一需要的 skill，不要整包全裝進核心。
  • 用 dummy project 測試訪談輸出與 PRD 品質。
  • 確認 skill 不會要求 token、cookie、帳密或客戶資料。
  • 通過測試後再寫成 Hermes 自己的規則。
不該做
  • 不讀內容就讓 agent 自動改本機 skill 目錄。
  • 把研究輸出直接當 Wiki 正式知識。
  • 讓未驗證 skill 直接處理 未公開或敏感資料。
  • 把 Grill Me 當成萬能保證；它只是降低模糊度，不保證成果完美。

可直接複製的提示詞

Grill Me 式需求訪談

在開始執行前，請先反過來訪談我。
請一次問一個最關鍵問題，直到你能整理出：
- 目標
- 使用者與場景
- 輸出形式
- 限制與不能做的事
- 成功標準
- 測試/驗收方式
在我回答完之前，不要開始實作。

訪談後轉 PRD

請把剛剛的訪談整理成 PRD。
格式包含：
1. 背景與目標
2. 使用情境
3. 功能需求
4. 非功能需求
5. 不做的範圍
6. 驗收條件
7. 風險與待確認問題

PRD 轉 TDD

請根據這份 PRD 產生 TDD 計畫。
請列出：
- 紅燈測試
- 綠燈標準
- 邊界條件
- 手動驗收步驟
- 需要截圖或瀏覽器檢查的地方

請在製作 Michael 教學筆記 HTML 前先訪談我。
你要問清楚：
