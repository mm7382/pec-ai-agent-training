Claude Design + Claude Code：從想法到介面、再到開發交接的閉環流程

這份教材不是在教你「怎麼畫一張漂亮圖」，而是在教你怎麼把需求、原型、視覺規格與後續開發接成同一條流程。重點是閉環，不是單次生成。

整理基礎：Anthropic Claude Design / Claude Code / MCP 官方資料 + 實際操作案例 2026-06-30
Course Framing

這套東西真正有價值的地方，是讓不會畫設計稿的人，也能做出可討論、可修改、可交接給工程的介面結果。

這份內容最重要的三條主線是：第一，用 /design 從 Claude Code 直接做出第一版原型；第二，用 /design-sync 把既有專案整理成可重用的 design system；第三，把設計變更再帶回程式工作流。這比單純生成一張畫面實際得多。

3核心工作流
/design從需求直接做原型
/design-sync把既有風格收斂成系統
2-way設計與程式雙向銜接
白話理解 三條主線 官方補充 流程圖 不懂程式怎麼用 實務場景 限制 檢查清單 來源

先用白話講清楚：Claude Design 到底是什麼？

你可以把它想成一個會跟你對話的設計工作台。你不是先開 Figma 慢慢拉元件，而是先用自然語言說「我要一個手機設定頁、上面有頭像、下面有幾個切換欄、底部有刪除帳號按鈕」，它先幫你做出第一版。接著你再改顏色、改版型、改細節，最後把這份設計交回 Claude Code 或開發流程。

一句話定義
Claude Design 最重要的不是「會畫圖」，而是它讓設計稿不會停在設計稿，而能繼續往開發走。
Anthropic 官方目前的說法也很一致：它可用來做設計、互動原型、簡報、microsite，並且能和 Claude Code 保持同步，這就是它和一般單點視覺工具最大的差別。

這個主題在解什麼痛點

原始痛點

文字想法很難變成具體畫面

PM 或工程師通常知道自己想要什麼流程，但不一定能很快畫出像樣的介面。

協作痛點

設計和開發常常斷開

做完原型只是開始，真正痛苦的是交接給工程之後又要重新翻譯一次。

解法

讓原型、視覺規格與程式互相可讀

這套工作流想做的是減少「重講一次」和「重做一次」的浪費。

工程價值

先討論對，再實作

先把方向做成可看、可比較、可修改的畫面，會比直接改程式安全很多。

把內容拆成三條主線，你就比較看得懂

主線 1

從 /design 直接做第一版原型

第一條主線先示範：在 Claude Code 裡輸入設計需求，讓系統先做出手機頁面或其他原型，再打開設計畫布進一步調整。

主線 2

用 /design-sync 把既有風格收斂成 design system

第二條主線的重點不是再畫一次，而是把既有專案樣式抽出來，之後新的畫面都能沿用同一套視覺語言。

主線 3

匯入現有 app，再把結果帶回程式

第三條主線展示：先把既有 app 介面做成高擬真原型，再做顏色和細節調整，最後回到程式流程接著改。

最容易被忽略的點
示範看起來像是在「做設計」，但真正重要的是最後那一步：設計狀態能不能回到開發流程。少了這一步，就只是在玩視覺原型。

官方文件補充後，這個主題更清楚了

官方資訊 代表什麼意思 對你有什麼用
Claude Design 可做 designs、interactive prototypes、presentations 等 它不是只拿來畫單頁 UI，也能做簡報、內部頁面和互動展示 Michael 教學筆記頁、提案頁、簡報頁都能納入
/design 可從 Claude Code 發起建立、編輯與同步 你可以從程式工作流開始，而不是一定要先打開設計工具 比較適合本來就常在 Claude Code 工作的人
/design-sync 可把 codebase、設計檔或既有資產變成 design system 風格不是每次重猜，而是能被抽成可重複使用的規則 團隊工具、產品線與教學頁更容易維持一致
官方建議先完成 design system 再大規模推廣 如果沒有先定好品牌和元件規則，輸出會功能正常但風格 generic 非常適合你現在的 Michael 教學筆記風格需求
可 export 為 HTML、PPTX、PDF，或 handoff 到 Claude Code 設計成果不是只能看，是真的可交接 更貼近內部教材、提案和工程 handoff

如果你不懂程式，最實際的使用方式是這樣

PM / 需求提出者

先把想法變成畫面

你不用先會畫設計稿，只要把目標、畫面區塊、使用者和要顯示的資訊講清楚，就能先產出一版可討論的介面。

  • 適合：團隊工具、儀表板、設定頁、流程頁
  • 好處：少掉大量抽象溝通
設計協作者

把方向修得更準

第一版通常只負責方向，之後再用對話、元件註解或直接編修去調整配色、字級、間距和畫面節奏。

  • 適合：快速探索 2 到 3 種方向
  • 好處：不用從空白畫布重新起手
工程或 Claude Code 使用者

接手時更少重翻譯

當設計成果可 handoff 回程式流程時，工程接到的不是一句模糊描述，而是比較完整的設計意圖和視覺結果。

  • 適合：已有 codebase 的專案改版
  • 好處：少走冤枉路、少返工

官方現在最推薦的操作順序，其實很務實

  1. 先準備 design system。 官方很明確：如果沒有先建好 design system，輸出雖然能用，但會比較 generic，不像你自己的品牌。
  2. 再開始做專案。 包含原型、dashboard、landing page、團隊工具畫面等。
  3. 補上下文。 可以給 screenshot、既有設計、codebase 或文件，讓結果更像你真的要的東西。
  4. 先看第一版，不要急著求完美。 官方也把第一版定位成 starting point，真正的價值在後面的反覆修正。
  5. 需要時 handoff 給 Claude Code。 這一步才是把設計和開發接起來的關鍵。

放到實務情境裡，最有價值的是這四種用法

團隊工具改版

先做畫面提案，再決定要不要改程式

像設備監控頁、權限頁、資料維護頁，先做出 2 到 3 個方向再開發，會比直接改畫面安全很多。

Michael 教學筆記頁

先探索版型，再落成 HTML

可以先試標題節奏、圖表配置、色塊區分，等方向穩了再用內容輸出 agent 或 Codex 做正式 HTML。

產品線一致化

把既有視覺風格收斂成系統

這對多個 agent 同時出內容特別有幫助，不然每次生成都像不同團隊做的。

既有 app / web 專案

把現有介面拉進來做高擬真改版草稿

這很適合「不是從零開始」的產品，先做局部改版驗證，再決定工程排程。

限制和邊界也要先講清楚

限制 代表什麼 實際建議
Beta / 研究預覽特性仍在變動 介面、指令、穩定度可能會調整 正式流程先小範圍試行
沒有 design system 時，輸出容易 generic 功能可用，但不像你的品牌與產品 先把 個人教學網站風格資產收斂起來
很大的 codebase 可能影響流暢度 官方也建議大型專案要更注意同步方式 先針對重要頁面或元件做局部驗證
多人同時編輯仍不算成熟 同時一起改容易混亂 先定一個 owner，再收斂版本
它不是正式前端驗收的替代品 再好的原型也不等於最終產品已經可上線 仍要做實機檢查、互動驗證與工程審查
最重要的一句提醒
不要把它當作「一鍵替代設計師」；比較實際的用法，是把它當成讓需求、原型、視覺規格和工程交接更順的協作層。

一頁檢查清單：如果你明天就想開始試，先做這些

  1. 先決定你要做的是新畫面探索、既有專案改版，還是 design system 建置。
  2. 如果是企業內部正式輸出，優先把品牌色、字體、元件規則整理成可上傳資產。
  3. 第一版只求方向，不要求完美，先把抽象需求變成可看畫面。
  4. 用對話做大改，用元件註解做局部改，用直接編修做細節微調。
  5. 確認最後成果能回到 Claude Code 或開發流程，否則它只是停在展示原型。
  6. 正式導入前，先找一個 個人教學網站頁面做小型試點，最容易看出價值。
