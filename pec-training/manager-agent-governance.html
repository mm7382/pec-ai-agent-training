02 / Core Concepts

核心觀念不是拿來收藏，是要變成可執行的規則。

先想清楚，再開始做

不要讓 agent 一收到任務就直接改東西。先讓管理者 agent 說清楚要做什麼、會影響誰、風險在哪。

怎麼用：例如要新增工具、改設定、刪資料，都先列計畫，再決定要不要執行。

把工作分給專門的 agent

管理者 agent 不需要自己做所有事。它負責分派，讓不同 agent 做自己擅長的部分。

怎麼用：查資料交給研究 agent，寫程式交給 coding agent，整理知識交給 wiki agent。

權限要分級

不是每個 agent 都能改檔、刪檔、裝工具。越危險的動作，越需要先問人。

怎麼用：看資料可以自動做；刪除、安裝、寄信、改核心設定，要先經過確認。

做完要檢查

agent 說做完，不代表真的成功。要看測試、錯誤訊息、服務狀態或畫面結果。

怎麼用：例如程式要跑語法檢查，網頁要能打開，bot 要確認有在運作。

自動整理交接

對話很長時，agent 容易忘記前面做過什麼。要定期整理成簡短交接紀錄。

怎麼用：寫清楚：完成什麼、改了哪裡、現在狀態、下一步做什麼。
03 / Implementation Plan

建議用階段推進，避免一次把 agent OS 改成難維護的大系統。

步驟 1

先由管理者判斷

這件事是聊天、查資料、寫程式、整理知識，還是高風險操作。

步驟 2

分派合適 agent

讓專門的 agent 做專門的事，管理者只負責協調與控風險。

步驟 3

危險動作先確認

安裝、刪除、寄信、改設定，都要先讓人確認。

步驟 4

做完留下紀錄

檢查結果、重要決策、下一步，都要寫進報告或知識庫。

04 / Review Checklist

每次 agent 進化後，用這份清單確認沒有留下分裂記憶或失控權限。

  • 這件事有沒有先判斷風險？
  • 有沒有分給最適合的 agent？
  • 危險操作有沒有先問人？
  • 做完有沒有真的檢查？
  • 有沒有留下簡短紀錄，方便下次接續？
05 / One Page Summary
管理 agent 的 agent 就像一個專案負責人。它不用自己做全部工作，但要知道誰適合做什麼、哪些事要先問人、做完要怎麼檢查、結果要記在哪裡。這樣多個 agent 才會像團隊，而不是一堆各自忙碌的工具。
