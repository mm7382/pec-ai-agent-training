Loop Engineering

Loop 不是讓 AI 自己跑

它是一個有邊界的系統：誰觸發、怎麼驗證、何時停止，都要先被設計清楚。

自動化之前，先系統化。
觸發 Prompt / 時間 / 事件
執行 讀上下文、改檔、調工具
驗證 測試、分數、審查、證據
停止 完成、到上限、需要人工
Trigger流程什麼時候開始。
Verify什麼證據算完成。
Stop什麼條件必須停。

四種 Loop 模式

不要照順序解鎖。先看任務需要交給系統哪一件事，再選最小可行模式。

1

Turn-based

人給一輪任務，Claude 執行、檢查、回報，人再決定下一步。

觸發：使用者 prompt
驗證：技能、測試、人工 review
停止：完成或需要更多上下文
2

Goal-based

把「完成」寫成可驗證目標，系統沒達標就繼續嘗試。

觸發：/goal 或明確目標
驗證：測試通過、分數達標、指標證據
停止：達標或達到最大嘗試次數
3

Time-based

同一個檢查定期重跑，適合外部狀態會改變的任務。

觸發：時間間隔或排程
驗證：PR、CI、部署、資料更新狀態
停止：取消、完成、期限到期
4

Proactive

由事件或排程啟動，適合明確、重複、可審核的工作流。

觸發：issue、表單、排程、告警
驗證：workflow、skills、review agent
停止：每個任務完成，風險點交給人

該交給系統哪一段？

交出檢查 探索型任務，仍由人決定下一輪。
交出停止條件 完成標準可量化，讓系統自己迭代。
交出觸發時機 需要定期或外部狀態變化後檢查。
交出整段流程 重複、明確、可審核，才適合主動化。

五個安全邊界

Loop 的價值不是跑更久，而是在明確規則內把人從重複調度中解放出來。

完成證據

測試結果、分數、截圖、log、審查清單。

停止規則

成功、失敗、上限、需要人工判斷。

成本上限

限制 turns、時間、模型等級與嘗試次數。

權限範圍

哪些工具可自動用，哪些一定要先問人。

人工關卡

發信、刪檔、部署、付款、客戶資料都保留人工確認。

開工前檢查

先回答三件事，再決定是否要讓 agent 進入 loop。

1. 觸發是否清楚？

  • 由人下 prompt？
  • 由時間排程？
  • 由 issue、CI、表單或告警觸發？

2. 驗證是否可量化？

  • 測試是否通過？
  • 分數是否達標？
  • 輸出是否能被 reviewer 檢查？

3. 停止是否明確？

  • 最多跑幾輪？
  • 失敗幾次要停？
  • 什麼情況必須交回人？
