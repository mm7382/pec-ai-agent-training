核心概念與名詞解釋

AI 第二大腦不是傳統筆記庫
傳統筆記工具多半是人讀的資料庫；AI 第二大腦要讓 agent 能讀、能分類、能提煉、能回寫成可用知識。
核心概念
  • Claude Code 的 memory 文件指出，CLAUDE.md 與 auto memory 會在對話開始時載入，適合放專案規則與偏好。
  • Claude Code 的 .claude directory 文件說明，專案與使用者層級可以放 instructions、settings、skills、subagents 與 memory。
  • 真正可用的第二大腦要分清楚：原始 inbox、整理後 knowledge、可執行 skills、素材 resources、以及 registry/index。
教學提示：實務上可以把產品資料、會議記錄、教材規則、圖片範例與 agent skills 放進同一套可檢索工作區。
Prompt 1：建立知識庫骨架
第一段提示詞的目標，是讓 Claude Code 在空資料夾裡建立第二大腦的目錄、規則、索引與基本工具。
核心概念
  • 重點不是一次產出很多內容，而是先有穩定結構：inbox、knowledge、skills、resources、registry、meta。
  • CLAUDE.md 要寫明 agent 如何處理新資料、如何分類、何時需要人工確認。
  • 第一版就要有簡單測試或檢查指令，避免只建立空資料夾。
教學提示：這段適合在 Claude Code 開新資料夾時直接貼上。
Prompt 2：建立本地指揮台
第二段提示詞的目標，是把資料流做成可視化面板，讓使用者看到 inbox、AI 加工、人工確認與入庫狀態。
核心概念
  • 指揮台不只是好看，核心是降低黑箱：每筆資料在哪個狀態、為什麼被分類、是否需要人工確認。
  • 面板應支援搜尋、標籤、狀態、來源類型與待處理清單。
  • 這可以讓第二大腦從資料夾變成可營運的 workflow。
教學提示：這段適合在骨架完成後貼給 Claude Code，讓它建立本地 HTML/Node dashboard。
Prompt 3：建立 Slack 入庫流程
第三段提示詞的目標，是讓手機看到的連結、圖片、視覺素材與靈感可以丟到 Slack channel，自動進入本地 inbox。
核心概念
  • Slack Events API 可以讓 app 或 bot 回應 Slack 活動；實作時可用 Socket Mode 或 HTTP endpoint。
  • 入庫流程要先保存 raw payload，再做分類與摘要，不要一開始就覆蓋原始資料。
  • 每筆資料都要進 registry，保留來源、時間、傳送者、檔案路徑與處理狀態。
教學提示：這段適合建立 brain-inbox channel 後使用；token 與 signing secret 要放在 .env，不要寫進文件。
Skill 型知識：讓 agent 學會做事
Claude Code skills 是一種可重複能力單元；第二大腦不能只存資料，也要存 agent 可以直接使用的工作方法。
核心概念
  • Claude Code skills 文件說明，skills 可延伸 Claude 能力，也可包含自訂指令與 bundled skills。
  • Anthropic skills repo 說明，skills 是 instructions、scripts、resources 的資料夾，讓 Claude 在特定任務表現更穩定。
  • 因此第二大腦要能判斷哪些內容只是資料，哪些值得沉澱成 skill。
教學提示：實務上可以把教材格式、PM issue triage、客戶回覆草稿、PR review 都整理成 skill，而不是每次重新寫 prompt。
實務導入方式
建議先用小範圍流程試行：教材素材入庫、產品資料整理、prompt/skill 沉澱，再擴到跨部門 agent workflow。
核心概念
  • 第一階段：只做本機資料夾與 registry，先把資料放對位置。
  • 第二階段：加 dashboard 與人工確認，避免 AI 自動亂分類。
  • 第三階段：加 Slack 入庫與 skill_candidate 分流，讓日常素材能被 agent 使用。
教學提示：先從教材產製流程試：每次來源素材、prompt、圖片範例與教學偏好都要能入庫，並回寫成可重複使用的 skill。

可複製提示詞

Prompt 1：建立知識庫骨架

你是我的 Claude Code 架構師。請在目前空資料夾建立一套「AI 第二大腦」本地知識庫骨架，目標是讓人和 agent 都能使用。

請建立以下結構：
- inbox/：所有未整理資料入口，支援 text、link、image、file 四類來源。
- knowledge/：整理後的 Markdown 知識，依 domain/topic/source 分類。
- skills/：可重複使用的 agent skills，每個 skill 都要有 SKILL.md、用途、觸發條件、步驟、輸入輸出與檢查項。
- resources/：圖片、視覺素材、參考圖與附件。
- registry/：索引檔，追蹤每筆資料的來源、狀態、標籤、owner、最後更新時間與是否已被 agent 使用。
- meta/：標籤系統、分類規則、命名規則、資料保留規則。

請同時產生：
1. CLAUDE.md：說明 agent 如何處理新資料、如何分類、如何避免重複、何時要問使用者。
2. README.md：給人看的使用說明。
3. scripts/healthcheck.mjs：檢查必要資料夾與 registry 是否存在。
4. 一份 sample inbox item 與整理後 sample knowledge item。

限制：
- 不要使用雲端資料庫。
- 不要刪除既有檔案。
- 所有文件使用繁體中文，工具名保留英文。
- 完成後請執行 healthcheck，並回報建立了哪些檔案。
用途：第一段提示詞的目標，是讓 Claude Code 在空資料夾裡建立第二大腦的目錄、規則、索引與基本工具。 這段適合在 Claude Code 開新資料夾時直接貼上。

Prompt 2：建立本地指揮台

請在目前 AI 第二大腦專案中建立一個本地「知識指揮台」。

目標：讓我用瀏覽器查看資料從 inbox 到 knowledge / skills / resources 的整個流程。

請建立：
1. public/dashboard.html：單頁 dashboard。
2. scripts/dashboard-server.mjs：本地 server，預設 port 8788。
3. scripts/scan-registry.mjs：掃描 inbox、knowledge、skills、resources，產生 registry/index.json。

Dashboard 必須包含：
- Inbox Queue：未處理資料清單。
- AI Processing：已被 agent 摘要、分類、去重的資料。
- Review Queue：需要人工確認的資料。
- Knowledge Library：已入庫 Markdown 知識。
- Skill Library：已整理好的 agent skills。
- Resource Library：圖片、視覺素材與附件。
- 每張卡片顯示 title、type、tags、status、source、updated_at。
- 支援依 type/status/tag 搜尋與篩選。

設計要求：
- 個人教學工具風格：白底、深灰文字、紅色主線、資訊密度高、手機也能看。
- 不做 landing page，第一畫面就是 dashboard。
- 不要使用外部 CDN。
- 完成後啟動 server，告訴我 localhost URL。
用途：第二段提示詞的目標，是把資料流做成可視化面板，讓使用者看到 inbox、AI 加工、人工確認與入庫狀態。 這段適合在骨架完成後貼給 Claude Code，讓它建立本地 HTML/Node dashboard。

Prompt 3：建立 Slack 入庫流程

請為目前 AI 第二大腦專案建立 Slack 入庫流程。

使用情境：我在手機看到文章、圖片、GitHub 連結、靈感素材時，丟到 Slack 的 #brain-inbox channel，資料要自動進入本機 inbox，等待 agent 整理。

請建立：
1. scripts/slack-inbox-bot.mjs：監聽 Slack Events API 或 Socket Mode。
2. .env.example：列出 SLACK_BOT_TOKEN、SLACK_APP_TOKEN、SLACK_SIGNING_SECRET、SLACK_CHANNEL_ID。
3. inbox/raw/：保存原始 Slack payload。
4. inbox/items/：保存整理後的 item JSON。
5. registry/index.json 更新邏輯。

每筆入庫資料要包含：
- id
- received_at
- slack_channel
- slack_user
- message_text
- links
- files
- inferred_type：text / link / image / file / skill_candidate
- suggested_tags
- status：new / processed / needs_review / archived

處理規則：
- 先保存 raw payload，再做加工。
- 不要把 Slack token 寫進任何文件。
- 圖片與檔案下載到 resources/slack/YYYY-MM-DD/。
- 如果內容像 GitHub skill 或 prompt，標成 skill_candidate。
- 完成後提供本機測試方式與啟動指令。
用途：第三段提示詞的目標，是讓手機看到的連結、圖片、視覺素材與靈感可以丟到 Slack channel，自動進入本地 inbox。 這段適合建立 brain-inbox channel 後使用；token 與 signing secret 要放在 .env，不要寫進文件。

知識分流與治理

Inbox 只收原始資料

手機丟進來的連結、視覺素材、prompt、GitHub skill 先進 inbox，保留 raw data，不要一開始就改寫。

內容重點研究依據

AI 先分流再加工

文字進 knowledge，圖片與附件進 resources，值得重複使用的方法進 skills，所有項目都寫入 registry。

內容重點研究依據

人工確認後再入庫

需要人工判斷的內容先進 review queue，避免 agent 把錯資料、重複資料或低品質 skill 直接納入正式知識庫。

實務導入方式

教材素材庫

落地：把來源連結、提示詞、圖片範例、個人教學版型規則與完成後 HTML 都進 registry，後續教材產製 agent 可重用。

Skill 候選區

落地：把 GitHub skill、內部 SOP、使用者確認過的 prompt 標成 skill_candidate，再整理成可被 agent 觸發的 SKILL.md。

手機入庫流程

落地：用 Slack channel 當行動入口，讓外出看到的連結或素材能先進本地 inbox，回到電腦後再由 agent 整理。
