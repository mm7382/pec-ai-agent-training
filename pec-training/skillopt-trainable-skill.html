SkillOpt：把提示詞工程變成可訓練的 Skill 系統

這份教材把 SkillOpt 拆成一個工程問題：skill 不再只是手工 prompt，而是 frozen agent 的外部可訓練狀態，能用 rollout、驗證集、受控編輯與部署產物來管理。

整理基礎：SkillOpt project / GitHub / arXiv 2026-06-24
Course Framing

這堂課要回答的不是「prompt 要怎麼寫更漂亮」，而是「skill 能不能像模型一樣被訓練、驗證、版本化」。

SkillOpt 的突破點在於：它不去碰模型權重，而是把 skill 文件本身當成可優化對象。對 ADLINK 這種會長期維護教學 HTML、研究報告、Codex SOP 與多 agent 流程的團隊來說，這件事的價值比單次 prompt 技巧更大。

SkillOpt 重新定義的，不是修辭，而是 prompt 的治理方式

傳統 prompt engineering 的問題不是完全沒效，而是太依賴個人經驗。你可以寫出一份能跑的 prompt，但很難知道它是不是已經接近最佳、是不是可穩定重複、是不是可以移轉到另一個 harness。SkillOpt 的回答是：把 skill 文件當成 agent 的外部狀態，像訓練模型那樣去優化它。

一句話定義
Fine-tuning 解決的是權重怎麼變；SkillOpt 解決的是在權重不變的前提下，工作協議怎麼變得更強。
核心觀點：不要再把 skill 當成一段寫完就算的提示詞，而要把它當成一個有評估集、有版本、有最佳產物的可訓練元件。

先把舊世界講清楚：為什麼手工 skill 會很快碰到天花板

舊做法

靠直覺寫

人或大模型憑感覺起草一份 skill，跑幾次不壞就收工。

核心缺陷

沒有優化方向

你不知道哪一句規則真的有用，也不知道哪一段只是看起來合理。

結果

可跑不等於夠好

同一任務可能 A 寫法 60 分、B 寫法 85 分，但傳統做法很少真的把兩者拉上驗證集比較。

SkillOpt 的解法

把 skill 變成訓練目標

重點不再是「這份 prompt 能不能跑」，而是「這份 skill 能不能在 held-out gate 上穩定變強」。

核心迴圈：Rollout、Reflect、Edit、Gate

SkillOpt 的 loop 刻意模仿深度學習。target model 與 harness 固定不動，只有 skill 文件會變。每一輪先用當前 skill 跑任務，蒐集成功與失敗軌跡，再由 optimizer model 提出小幅編輯，最後用 held-out validation 決定要不要接受。

重點不是讓 optimizer 每次整份重寫。SkillOpt 的穩定性來自「受控小改」，不是來自「一次大翻修」。

真正讓它像訓練而不是像亂改 prompt 的，是這些控制項

Textual Learning Rate

論文把 edit budget 視為文字空間的 learning rate。一次只允許有限次 add / delete / replace，避免把本來有效的規則一起洗掉。

Rejected-Edit Buffer

被 gate 擋下的候選編輯不直接消失，而是變成負面經驗，讓下一輪 optimizer 避開類似失敗方向。

Slow / Meta Update

除了步進式 edits，還保留較長週期的總結與穩定規律，避免 skill 只會追逐短期 gain。

Separate Optimizer Model

寫 skill 的模型與執行任務的 target model 分工，讓「反思怎麼改」與「照規則執行」這兩種能力不要互相干擾。

工程上的意思很直接：如果沒有受控步長、沒有 reject memory、沒有 gate，你做的就不是訓練，而比較像高級版 prompt 瞎猜。

結果不是小修小補，而是把 skill 這層證明成有效的 domain adaptation

最有營運價值的地方：best_skill.md 可以轉移

論文與專案頁都強調，部署時 target model 不需要帶 optimizer memory，只需要最終的 best_skill.md。而且這份 skill 不只對原本的 target model 有用，還能跨模型尺寸、跨 Codex / Claude Code harness、甚至對相近 benchmark 帶來正向效果。

對企業的意義

  • 可以集中在較強算力環境做 skill 訓練。
  • 訓練好後，把產物分發到不同 agent 與不同模型。
  • 不用每個場景都重新 fine-tune 或重新手寫 prompt。

對個人的意義

  • 你維護的不再是很多段 prompt，而是一組經過驗證的 skill 資產。
  • 真正值錢的能力會慢慢從「很會寫 prompt」轉向「很會設 benchmark、很會做 gate」。
  • Agent engineering 開始更像 ML engineering。

對 ADLINK 最直接的落地方式

場景現在常見做法SkillOpt 思路下應改成什麼
教學 HTML 產線 每次重寫 prompt，靠人工看順不順 固定 lesson brief 結構、版面規則、禁用詞與 mobile gate，定期用評估樣本訓練教學 skill
Codex / Claude Code SOP 手工維護長指令，靠記憶提醒 把任務拆成可評測 skill，量化成功率、回合數、工具失敗率與格式一致性
研究報告 每篇報告都重講來源標準與保守語氣 把引用規則、可信度排序、禁止過度結論化等行為封裝成可驗證 skill
PM / triage 靠人手判斷 priority 與 owner 把分類標準、缺資訊時的追問、不可自動派工條件放進 skill，再用 held-out case 做驗證
如果只是把更多規則塞進 prompt，這還不叫 SkillOpt。真正的改變是：你開始有驗證集、接受條件、被拒絕的編輯記錄，以及可以分發的最佳 skill 產物。

SkillOpt 思路

定義任務與評估集
rollout 收集軌跡
optimizer 提出小幅 edits
held-out gate 決定是否接受
輸出可部署的 best_skill.md
對使用者來說，這代表未來你可能不需要一直手調 prompt；對團隊來說，這代表真正要投資的是 skill 設計、評估樣本與驗證流程。

如果你要在自己的 agent 系統借鏡 SkillOpt，先過這張清單

  1. 先挑一個高重複、可評分的工作流。
    例如教學 HTML、資料整理、ticket triage、PR review。不要一開始就選超開放、沒有驗收標準的任務。
  2. 先定義評估集，不要先堆 prompt。
    SkillOpt 成立的前提，是你知道什麼叫成功、什麼叫退步。
  3. 把 skill 當外部狀態，而不是把所有希望寄託在模型本身。
    模型可以固定，skill 可以迭代，這正是成本與可控性的關鍵。
  4. 限制每輪修改幅度。
    大改容易把既有有效規則一起改壞，小步編輯比較適合長期收斂。
  5. 保留 rejected edits 與驗證紀錄。
    這些不是失敗垃圾，而是之後避免重犯的負面樣本。
  6. 部署時只帶最佳產物，不要把訓練過程一起背上線。
    best_skill.md 應該是乾淨、可稽核、可分發的最終協議。
