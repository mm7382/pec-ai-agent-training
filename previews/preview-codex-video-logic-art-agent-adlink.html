3

核心概念與名詞解釋

從 Benchmark 轉向真實工作流驗證
新模型是否有價值，要用既有產線裡真正難、真正會痛的問題來測。
核心概念
  • 官方 benchmark 與案例可以作為參考，但實際使用體驗才決定是否值得投入。
  • 作者選擇自己長期使用的影片生產流程作為測試場景，而不是從零開始做 app demo。
  • 從零專案沒有舊程式碼與歷史包袱，較難看出模型處理真實複雜度的能力。
  • 真實工作流包含語音辨識、自動剪片、字幕、片頭、b roll、贅詞移除、縮圖與上架文案等多步驟。
把 Agent 當資深同事，不是當指令執行器
高能力 agent 的價值在於協作判斷與主動發現問題，而不只是照步驟執行。
核心概念
  • 過去使用 AI 常是下指令、檢查、修正、再循環，人的角色卡在每一步監工。
  • Fable-5 後，人的重點變成確認 agent 做的是對的事情、方向沒有歪掉。
  • 使用方式要像跟同事討論：請它提出建議、切法、做法與潛在問題。
  • 作者觀察到 agent 主動提出資料保護、驗證深度、PR 拆法、case file 切法等問題。
大目標必須搭配邊界、驗收與停損
任務可以放大，但 prompt 必須明確定義終點、驗收標準、禁區與停損條件。
核心概念
  • 高能力模型可直接接受較大的 goal，讓它自行找到路徑。
  • 只給一句目標不夠，必須說清楚什麼情況算完成、何時要停下來問人。
  • 禁區包含不要碰的資料、不要改壞的舊功能、不要刪除的檔案。
  • 模型越強，越要講清楚設計邊界，否則只是更快把錯的事情做完。
在乾淨工作區中執行長任務
讓 agent 長時間自主工作前，先用版本控制與隔離工作區降低破壞風險。
核心概念
  • 作者在實測前建立新的 Worktree，避免污染原本運作良好的主程式。
  • Worktree 被用來隔離新功能或修復任務，概念上像從 main 拉出乾淨工作區。
  • Auto Mode 的好處是不用每一步都按確認，適合明確邊界後的長任務執行。
  • 前期討論可用 Play Mode，確認清楚後再切到 Auto Mode 開工。
用機制層修復 Corner Case
資深工程師標準不是把輸出修到看起來對，而是找根因、改機制、補測試。
核心概念
  • retake 拼接難在多段重錄之間找到正確片段，並讓文字、語域、音量、底噪與停頓自然銜接。
  • 作者把壞掉的 case 收集下來，作為考驗模型的真實測試集。
  • 修問題的鐵律是修在機制層，不是頭痛醫頭、腳痛醫腳。
  • 以前的模型容易只修輸出結果，甚至不改程式；這不符合資深工程師標準。
雙層驗收：程式測試加人工判斷
AI 可以自動化大量工作，但品質判斷仍要結合測試與人的感官驗收。
核心概念
  • 第一層驗收是人工聽五個修正後的音檔，確認原始剪接問題是否消失。
  • 第二層驗收是跑測試程式，確認五個修正片段都 passed。
  • 作者也執行完整測試，確保原本正常的功能沒有被改壞。
  • 人的眼睛與耳朵仍負責界面、聽感與成果價值的判斷。
成本窗口與任務選擇策略
高成本模型應優先用在很長、很大、很複雜且過去卡住的任務。
核心概念
  • Fable-5 在特定訂閱方案中有時間限制，之後需透過 API 或 usage credit 支付。
  • 來源提到價格為 input 每百萬 10 美金、output 每百萬 50 美金，是 Opus 的兩倍。
  • 作者建議趁免費窗口挑一個真正卡很久、以前覺得 AI 做不到的任務測試。
  • 神級模型特別適合長、大、複雜任務，而不是瑣碎小任務。
