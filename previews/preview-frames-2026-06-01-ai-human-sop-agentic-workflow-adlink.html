AI 時代非技術人最該學的設計能力：把 Human SOP 變成 Agentic Workflow

Report Brief

這堂課要解決的問題

  • 很多非技術角色已經會請 AI 幫忙寫文案、整理資料或回答問題，但還停在單次 prompt；真正的差距在於能不能把人的工作 SOP 設計成 agent 可以穩定執行的 workflow。
  • Human SOP 通常寫給人看，會省略判斷條件、例外處理、輸入格式、確認點與錯誤回報；Agentic Workflow 則必須把這些隱性判斷轉成明確規則。
  • 本課重點不是寫程式，而是學會設計一套讓 AI agent 能執行、能追問、能檢查、能迭代修正的工作流程。
  • 教材輸出要抓住來源內容主軸：從人的 SOP 出發，拆成任務、輸入、判斷、工具、輸出、追問與回饋迴圈。

教學對象

  • PM / 業務 / 行銷 / 營運：平常有固定工作流程，但不會寫程式，想把流程交給 AI agent 協助執行的人。
  • 內部 agent owner：需要把需求說清楚，讓工程師或 Codex 能把 SOP 轉成可運作 agent 的人。
  • 系統工程師與自動化導入者：需要從使用者口中的流程，萃取成 agent workflow、工具權限與驗收規則的人。
  • 不適合對象：只想問 AI 一次答案、不願意整理流程、不想定義輸入輸出與確認規則的人。
Lesson Modules

教學重點模組

Visual Roadmap

Human SOP 到 Agentic Workflow 架構

01Layer 1

Human SOP - 先寫出人現在怎麼做，包含觸發條件、使用資料、判斷方式、交付物與誰要確認。

02Layer 2

Task Decomposition - 把 SOP 拆成 agent 可執行的小任務，例如讀取資料、分類、摘要、比對、產出草稿、提出問題。

03Layer 3

Decision Rules - 把人腦中的判斷條件寫出來，例如何時用 normal 模式、何時需要進一步查資料、何時必須問使用者。

04Layer 4

Tool & Context - 定義 agent 可以用哪些工具、讀哪些資料、輸出到哪裡，以及不能碰哪些權限。

05Layer 5

Feedback Loop - 第一版 SOP 一定會有問題，因此要設計錯誤回報、人工確認、修正紀錄與下一版 workflow。

System Workflow

工程師操作模型

STEP 1Step 1

訪談使用者 - 先問現在人工怎麼做、用哪些資料、最後交付什麼、主管或客戶如何判斷好壞。

STEP 2Step 2

整理 Human SOP - 用自然語言寫出觸發條件、輸入資料、操作步驟、判斷條件、輸出格式與確認點。

STEP 3Step 3

拆成 Agent Tasks - 將流程拆成讀取、理解、分類、產出、檢查、追問、回報等 agent 可執行單元。

STEP 4Step 4

補上例外處理 - 定義資料不足、格式錯誤、衝突資訊、權限不足、結果不可信時 agent 要怎麼處理。

STEP 5Step 5

加入 Clarifying Questions - 規定 agent 在資訊不足時必須先問 2 到 3 個問題，而不是硬產生看似完整的答案。

STEP 6Step 6

設計驗收與迭代 - 每次執行後紀錄成功、錯誤與人工修改，回寫到下一版 SOP 與 prompt。

Engineering Applications
1

SOP Parser

把人的流程文字轉成任務節點，避免 agent 只看到一段模糊需求。

2

Decision Gate

每個節點都要有判斷條件，例如直接執行、先查資料、先問使用者、或停止等待人工確認。

3

Context Pack

把必要背景、資料來源、角色、限制與輸出格式打包給 agent，降低胡亂補內容的機率。

4

Clarification Loop

當資訊不足時，agent 必須先問問題，補齊再繼續，而不是產生空泛結果。

5

Review Log

保留每次輸出、錯誤、人工修改與採納結果，讓 workflow 能持續變準。

Teaching Notes

核心觀念

  • 非技術人最該學的不是程式語法，而是 workflow design：把工作的前因後果、判斷邏輯與驗收標準說清楚。
  • 不要在 SOP 裡寫死單一路徑；agent 需要知道什麼情況可以直接做、什麼情況要切換模式、什麼情況必須先追問。
  • Agent 不該只產出答案，也要能產出 2 到 3 個釐清問題，幫使用者補足缺失的背景、資料與限制。
  • 好的 Agentic Workflow 要讓錯誤可見：如果結果不對，要能回頭知道是哪個輸入、判斷、工具或輸出規則出問題。
  • 第一版 workflow 的目標不是完美，而是可測試、可修正、可複製；每次使用後都要把問題寫回 SOP。

ADLINK 可落地場景

  • art agent 教材流程：來源素材 → 內容轉錄 → 重點視覺素材 → ADLINK HTML 預覽 → 檔名確認 → 位置確認 → 是否寄信。
  • PM Issue Summary：讀取專案資料 → 找出 critical issue → 分類 RF/ME/Thermal/Safety → 產生 HTML → 人工確認 → 寄出。
  • 會議紀錄整理：輸入 transcript → 萃取決議、風險、owner、due date → 產出追蹤表 → 釐清缺失資訊。
  • 教學文件產線：把工程師口述 SOP 轉成教學章節、截圖需求、驗收清單與可重複 prompt。
  • Agent 需求規格：非技術人先用這套方法寫出 workflow，再交給 Codex 或工程師實作工具與自動化。

導入風險與控制

  • SOP 太模糊：如果只有「幫我整理資料」，agent 會自行腦補；必須補上輸入、輸出、判斷條件與例外處理。
  • 權限太大：agent 可以讀檔、寄信或修改資料時，必須保留人工確認點，避免錯誤自動擴散。
  • 輸出無法驗收：每個 workflow 都要定義完成標準，例如格式、欄位、來源、截圖、是否需要寄信。
  • 第一版不可靠：課程重點提醒第一版 SOP 一定會有問題，因此要把修正流程設計進去。
  • 缺少追問機制：資訊不足時若不追問，agent 會產生看似合理但其實錯的內容。

常見誤解

  • 誤解 1：非技術人不能設計 agent。實際上，非技術人最懂工作流程，只要把 SOP 說清楚，就能成為 agent 設計起點。
  • 誤解 2：Agentic Workflow 等於自動化全部流程。正確做法是先定義哪些步驟可自動，哪些步驟要人工確認。
  • 誤解 3：寫一次 SOP 就完成。第一版一定會暴露問題，真正價值在於持續迭代。
  • 誤解 4：越詳細越好。詳細但沒有判斷規則、輸入輸出與例外處理，仍然不是可執行 workflow。

工作坊設計

  • 0-10 min：選一個真實工作流程，例如教材製作、會議整理、PM issue summary 或客戶回覆草稿。
  • 10-25 min：寫出 Human SOP，包含觸發條件、資料來源、人工判斷與交付物。
  • 25-45 min：拆成 Agentic Workflow 節點，標記每一步的輸入、工具、輸出與 decision gate。
  • 45-60 min：加入 2 到 3 個釐清問題與錯誤處理規則，避免 agent 硬做。
  • 60-75 min：用一個測試案例跑流程，記錄哪一步資訊不足、哪一步輸出不準。
  • 75-90 min：把修正寫回 SOP，形成第二版 workflow。

驗收清單

  • 能把人的 SOP 拆成 agent 可執行的任務節點。
  • 每個節點都有輸入、處理方式、輸出與判斷條件。
  • 資訊不足時，agent 會先問釐清問題，不會直接捏造答案。
  • 流程中有人工確認點，尤其是寄信、對外輸出與修改正式資料前。
  • 每次執行後會保留錯誤與修改紀錄，回寫成下一版 workflow。
  • HTML 教材能把來源內容整理成教學模組，並用代表性視覺素材輔助學員理解。
